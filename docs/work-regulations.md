# Task-трекер
В качестве task-трекера используем GitHub Projects.

В столбце `Pre-proccess Backlog` лежат задачи по подготовке к разработке, которые на данный момент не приняты в работу.  
В столбце `Backlog` лежат задачи, которые на данный момент не приняты в работу.  
В столбце `Ready` лежат назначенные задачи, к выполнению которых разработчик еще не приступил.  
В столбце `In progress` лежат задачи, которые на данный момент выполняются разработчиком.  
В столбце `In review` лежат задачи, на которых создан Pull Request и которые ожидают ревью.  
В столбце `Testing` лежат задачи, которые на данный момент проходят тестирование.  
В столбце `Done` лежат выполненные задачи.  

По мере изменения статуса задачи разработчику нужно передвигать карточку задачи по столбцам. Рекомендуется делать это своевременно.

# Порядок работы над задачами
Как только задача была взята вами в работу следуют в системе контроля версий Git создать свою ветку от ветки `dev`.
```
git checkout dev # если вы не на ней
git pull origin dev
git checkout -b <название вашей ветки>
```

## Правила именования веток
Название ветки начинается с префикса, обозначающим типа изменений, которые вносит ветка.
### Основные типы изменений:
- `feature` - добавление нового функционала
- `bugfix` - исправление ошибки
- `release` - подготовка релиза
- `test` - изменения в тестах

Следом за типом изменений, которые вносит ветка следует символ `/`, после которого указывается ваше название. В названии слова должны разделяться символом `-`. Название должно кратко описывать, что сделано в этой ветке. Например `add-login-function`.
Итоговое название такой ветки будет следующим: `feature/add-login-function`.

## Правила создания коммитов
```
git add .
git commit -m "<название коммита>"
```

Коммит - логическая точка подзадачи.
Коммиты следует именовать следующим образом:
`<тип изменений>(<область ваших изменений>): <ваше сообщение>`.

### Основные типы изменений:
- `feat` - новый функционал
- `fix` - исправление ошибки
- `style` - форматирование кода
- `refactor` - рефакторинг
- `test` - изменения в тестах
- `chore` - прочие изменения, например, настройка билда, изменения в конфигурационных файлах и т.д.

В качестве области ваших изменений стоит указывать название модуля приложения, в котором разработчик работал.
Сообщение коммита должно быть осмысленным и информативным.


## Правила создания PR (Pull Request)
После того как вы запушили изменения в удаленный репозиторий и убедились, что все работает как надо необходимо создать Pull Request вашей ветки в ветку `dev`.  
В названии пулл реквеста кратко указываем, что было сделано (стараемся на русском языке). В описании подробно описываем, что было сделано и вставляем ссылку на задачу в Task-трекере. Не забываем назначать PR на себя (Assignees).
